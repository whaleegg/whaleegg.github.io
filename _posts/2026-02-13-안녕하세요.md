이 글은 이 블로그를 실제로 구축하면서 겪은 과정과 문제 해결 흐름을 처음부터 끝까지 정리한 기록이다. 단순히 테마를 한 번 적용해 본 수준이 아니라, 로컬 실행 환경 복구, 이미지 자산 구조 정리, Open Graph 이미지 설정, 사이드바 배경 적용, GitHub 배포까지 한 번에 정리했다.

아래 스크린샷들은 이 글을 작성하면서 로컬 빌드 결과를 기준으로 직접 캡처한 화면이다.

## 시작 상태

처음 저장소를 열었을 때는 이미 Chirpy 기반 구조는 들어와 있었다. `_posts/`, `_layouts/`, `_includes/`, `_sass/`, `_javascript/` 같은 기본 구조도 있었고, 샘플 포스트도 몇 개 들어 있었다.

문제는 "블로그가 동작하는가"와 "앞으로 운영 가능한 상태인가"는 별개의 이야기였다는 점이다. 실제로 확인해 보니 다음과 같은 작업이 필요했다.

- 로컬 Jekyll 서버 실행 환경 정리
- `_config.yml`의 소셜 링크와 기본 메타데이터 점검
- 이미지 자산 구조 정리
- 샘플 포스트의 깨진 이미지 복구
- 블로그 운영 규칙 문서화
- GitHub에 다시 안전하게 배포할 수 있는 상태 만들기

## 1. 로컬 Jekyll 실행 환경 복구

가장 먼저 막힌 부분은 로컬 실행이었다. `bash tools/run.sh`를 실행했더니 `bundle: 명령어를 찾을 수 없음` 오류가 났고, 그다음에는 Ruby 버전 충돌과 Bundler 충돌이 이어졌다.

확인해 보니 이 저장소가 사용하는 Chirpy 의존성은 Ruby `~> 3.1` 이상을 요구하고 있었는데, 로컬 환경은 Ruby `3.0.2`였다. 그래서 단순히 `bundle install`만으로는 해결되지 않았다.

결국 다음 순서로 정리했다.

1. `rbenv`와 최신 `ruby-build` 설치
2. Ruby `3.2.11` 설치
3. Bundler `2.5.23` 설치
4. 예전에 잡혀 있던 `GEM_HOME="$HOME/gems"` 설정 제거

특히 마지막이 중요했다. 처음에는 Ruby는 `rbenv` 경로를 쓰고 있었는데, gem 경로는 예전 설정 때문에 `~/gems`를 참조하고 있어서 Bundler가 꼬였다. 이 부분을 정리한 뒤에야 `bundle install`과 `bundle exec jekyll s -l -H 127.0.0.1`이 정상 동작했다.

이 과정을 거치면서 지금 이 블로그의 로컬 실행 기준은 다음으로 정리됐다.

```bash
cd /home/whaleegg/blog/whaleegg.github.io
bundle exec jekyll s -l -H 127.0.0.1
```

환경 재현성을 위해 `.ruby-version`도 저장소에 포함했다.

## 2. `_config.yml` 정리

로컬 서버가 뜬 뒤에는 `_config.yml`을 점검했다. 가장 먼저 보였던 문제는 footer의 GitHub 링크였다.

사이드바 하단 GitHub 아이콘은 정상적으로 내 계정으로 연결됐지만, 페이지 맨 아래 footer의 저작권 링크는 `https://github.com/username` 같은 placeholder를 그대로 쓰고 있었다. 알고 보니 이 둘은 같은 설정을 보지 않았다.

그래서 `_config.yml`에서 다음 항목들을 정리했다.

- `social.links[0]`를 실제 GitHub 계정으로 수정
- `avatar` 경로 정리
- 기본 `social_preview_image` 설정
- `cdn` 비활성화

처음에는 `social_preview_image`만 추가하면 될 줄 알았는데, `cdn` 설정이 켜져 있으면 모든 미디어 경로 앞에 CDN URL이 붙는 구조였다. 그 결과 기본 OG 이미지가 실제 저장소 경로가 아니라 엉뚱한 CDN 주소를 바라보는 문제가 생겼다.

이 문제를 해결한 방식은 두 가지였다.

- `cdn`을 끈다
- `social_preview_image`는 절대 URL로 준다

최종적으로는 `cdn`을 비활성화하고, `social_preview_image`를 절대 URL로 지정했다.

## 3. 이미지 자산 구조 정리

블로그를 운영하다 보면 글마다 이미지가 쌓인다. 그래서 중간에 이미지 구조도 함께 정리했다.

최종적으로 잡은 기준은 다음과 같다.

- 사이트 공통 이미지: `assets/img/site/`
- 여러 글에서 재사용하는 이미지: `assets/img/commons/`
- 글별 이미지: `assets/img/posts/YYYYMMDD-slug/`

이 구조를 택한 이유는 단순하다.

- 글과 이미지의 관계를 쉽게 찾을 수 있다
- 파일명 충돌을 줄일 수 있다
- Git으로 추적하기 쉽다
- 외부 CDN 의존성을 줄일 수 있다

실제로 이 과정에서 다음 파일들도 재배치했다.

- `profile.jpg` -> `assets/img/site/profile.jpg`
- `background.jpg` -> `assets/img/site/background.jpg`
- `og-default.png` -> `assets/img/site/og-default.png`

이후 새 글을 쓸 때는 가능한 한 `/assets/img/...` 절대경로를 사용하기로 정했다.

_카테고리 화면도 함께 확인하면서 구조가 의도대로 동작하는지 점검했다._

## 4. 깨진 샘플 포스트 이미지 복구

Chirpy 샘플 포스트 중 하나인 `Text and Typography` 글은 이미지가 깨져 있었다. 원인은 이 글이 외부 데모 경로를 기준으로 이미지를 참조하고 있었기 때문이다.

처음에는 "이미지를 저장소에 넣으면 끝나겠지"라고 생각했는데, 실제로는 `cdn` 설정까지 같이 영향을 주고 있었다. 이미지 파일을 로컬에 넣어도 생성된 HTML에는 여전히 CDN URL이 붙고 있었고, 그래서 브라우저는 다른 곳에서 이미지를 찾고 있었다.

정리 과정은 이랬다.

1. 데모용 이미지 파일을 저장소로 가져옴
2. `assets/img/commons/`, `assets/img/posts/...` 구조에 맞게 배치
3. 포스트 안의 이미지 경로를 `/assets/img/...` 기준으로 수정
4. `cdn` 비활성화
5. Jekyll 서버 재시작

여기서 한 번 더 배운 점은, `_config.yml` 변경은 자동 리로드만으로는 바로 반영되지 않을 수 있다는 점이다. 실제로 `cdn`을 껐는데도 생성된 HTML에는 예전 경로가 남아 있었고, 서버를 재시작한 뒤에야 정상 반영됐다.

## 5. Open Graph 이미지와 배포 후 확인

블로그 글은 단순히 화면에 보이는 것만 신경 쓰면 끝이 아니다. 링크를 공유했을 때 어떤 썸네일과 설명이 뜨는지도 중요하다.

그래서 기본 Open Graph 이미지도 따로 넣었다. 포스트마다 대표 이미지를 따로 두지 않는 경우에도 공유 미리보기가 비어 보이지 않게 하려는 목적이었다.

처음 추가한 이미지는 비율이 조금 아쉬웠지만, 우선 기본 이미지로 연결해 두고 나중에 필요하면 전용 OG 이미지를 새로 만들기로 했다. 중요한 점은 일단 사이트 전역에 기본 `og:image`가 들어가도록 만든 것이다.

그리고 실제로는 브라우저에서 "페이지에 이미지가 안 보인다" 싶을 때도 있었다. 확인해 보니 코드와 배포는 정상인데, 브라우저 캐시 때문에 예전 CSS가 잡혀 있는 경우가 있었다. 이때는 강력 새로고침이 필요했다.

## 6. 사이드바 배경 이미지 적용과 최적화

블로그의 분위기를 조금 더 만들고 싶어서 사이드바에 배경 이미지를 적용했다. 다만 본문 전체 배경으로 바로 깔면 가독성이 떨어질 수 있어서, Chirpy 구조에 맞게 사이드바에만 적용했다.

스타일은 다음 방향으로 잡았다.

- 배경 이미지는 사이드바에만 사용
- 위에 어두운 그라데이션 오버레이 추가
- 텍스트와 아이콘은 배경 위에서 읽히도록 대비 확보

적용 자체는 금방 됐지만, 또 다른 문제가 있었다. 원본 이미지 용량이 너무 컸다. `4000x3000`, 약 `4.1MB` 정도라서 배경용으로는 과했다.

그래서 원본은 그대로 보관하고, 별도로 최적화본을 만들었다.

- 원본: `assets/img/site/background.jpg`
- 최적화본: `assets/img/site/background-optimized.jpg`

최적화본은 `1800x1350`, 약 `681KB` 수준으로 줄였고, 실제 CSS는 최적화본을 바라보도록 바꿨다. 이렇게 하면 원본은 보존하면서도 운영 성능은 개선할 수 있다.

## 7. Git 설정과 GitHub 배포

로컬에서 다 확인한 뒤에는 Git 커밋과 배포까지 정리했다. 이 과정에서도 몇 가지 손볼 부분이 있었다.

먼저 저장소에 Git 작성자 정보가 설정돼 있지 않아서, 로컬 저장소 기준으로 사용자 이름과 이메일을 잡아줬다. 그 다음에는 HTTPS 원격 대신 SSH 방식으로 푸시할 수 있도록 정리했다.

SSH 키는 이미 로컬에 있었고, GitHub 계정에도 등록돼 있는 상태였다. 그래서 원격 주소를 SSH로 바꾸고 나니 이후 푸시는 훨씬 매끄러워졌다.

배포 과정에서는 이미 원격 `main`에 Dependabot PR이 먼저 들어와 있어서 바로 push가 거절되기도 했다. 이때는 fetch 후 rebase해서 정리한 다음 다시 push했다.

결과적으로 다음과 같은 변경이 정상적으로 원격에 반영됐다.

- 로컬 Ruby/Jekyll 실행 환경 정리
- `_config.yml` 메타데이터 및 링크 정리
- 이미지 자산 구조 정리
- 기본 OG 이미지 설정
- 샘플 포스트 이미지 복구
- 사이드바 배경 이미지 추가 및 최적화

## 8. 블로그 운영 규칙 문서화

마지막으로, 앞으로 블로그를 운영하면서 매번 같은 설명을 반복하지 않도록 `AGENTS.md`를 정리했다.

여기에는 다음과 같은 기준을 넣었다.

- 새 포스트 파일명 규칙
- `front matter` 기본 형식
- 카테고리/태그 운영 기준
- 이미지 저장 위치 규칙
- `_config.yml`, `_layouts/`, `_includes/` 수정 시 검증 우선순위
- 리뷰할 때 무엇을 먼저 볼지

이 문서가 있으면 이후에는 새 글 작성, 포스트 다듬기, 배포 전 점검을 더 일관되게 진행할 수 있다.

## 마무리

정리해 보면, 이 블로그를 구축하는 과정은 "테마를 설치했다"에서 끝나는 일이 아니었다. 실제로는 개발 환경, 설정 파일, 이미지 구조, 렌더링 결과, 배포 흐름까지 한 번씩 다 만져봐야 비로소 운영 가능한 상태가 됐다.

이번에 정리하면서 특히 중요하다고 느낀 점은 세 가지다.

- 로컬 실행 환경은 버전 매니저 기준으로 깔끔하게 정리할 것
- 이미지와 메타데이터는 처음부터 구조를 잡아둘 것
- `_config.yml` 같은 핵심 설정은 바뀌면 반드시 실제 생성 결과까지 확인할 것

이제는 단순히 "블로그가 있다"는 상태를 넘어서, 실제로 글을 계속 쓸 수 있는 운영 기반이 갖춰졌다고 볼 수 있다. 다음 단계는 이 구조 위에 실제 공부 기록과 프로젝트 글을 쌓아가는 일이다.
